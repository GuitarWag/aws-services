IAM Deep Dive

Identity & Access Management — concepts, policy types, and how AWS compares to GCP and Azure.

Source: AWS IAM User Guide

A — Core Concepts
№ 01 Identity

Principal

The who making the request

Any entity that can authenticate and make requests: a root user, IAM user, IAM role, federated user, or an AWS service acting on your behalf.

Root · IAM User · IAM Role · Federated · AWS Service
№ 02 Verification

Authentication

Prove you are who you claim to be

Verify identity via password+MFA (console), access keys (CLI/SDK), or temporary STS tokens. Federated users authenticate externally via SAML 2.0 or OIDC.

Password · Access Keys · STS Token · SAML · OIDC
№ 03 Decision

Authorization

Should this action be allowed?

AWS evaluates all applicable policies. Default is implicit deny. Any explicit Deny wins. An Allow is needed on both sides for cross-account access.

Implicit Deny · Explicit Deny · Explicit Allow
№ 04 Rules

Policy

The rulebook — a JSON document

Defines permissions: Effect (Allow/Deny), Action (which API calls), Resource (which ARNs), and optional Condition keys.

Effect · Action · Resource · Condition
№ 05 Delegation

Role

A temporary hat, not a person

Not a user — no username or password. A set of permissions any trusted principal can assume temporarily. Eliminates long-lived credentials. EC2, Lambda, and cross-account users all use roles.

AssumeRole · Temp Credentials · STS · Instance Profile
№ 06 Gatekeeper

Trust Policy

Who is allowed to assume this role?

A resource-based policy on the role itself. Specifies which principals (accounts, services, identity providers) are trusted to call sts:AssumeRole. Without it, no one can use the role.

sts:AssumeRole · Principal · Condition · ExternalId

C — AssumeRole / Temporary Credentials

The mechanism that replaces long-lived credentials with short-lived, auto-rotating ones. Used by EC2, Lambda, cross-account access, and federated users.

01
Principal calls sts:AssumeRole
Provides the ARN of the role it wants. E.g. EC2 instance requests its attached instance profile role.
02
STS reads the Role's Trust Policy
Checks if the calling principal is a trusted entity. If not listed → AccessDenied immediately.
03
STS issues temporary credentials
Returns AccessKeyId + SecretAccessKey + SessionToken. Valid 15 min – 12 hr (default 1h).
04
Principal uses creds for API calls
All calls now come from the Role's identity. AWS evaluates the role's identity-based policies for each action.
05
Credentials expire automatically
No revocation step needed — they just stop working. To continue, the principal calls AssumeRole again.
Key facts
  • Roles ≠ Users. A role has no password or long-lived access keys. It's a set of permissions + a trust policy.
  • Cross-account access works by having the target account's role trust the source account's principal — no credential sharing.
  • EC2 Instance Profiles wrap a role and attach it to an EC2 instance. The SDK fetches credentials automatically from the metadata endpoint (169.254.169.254).
  • ExternalId is a condition in the Trust Policy that prevents the confused deputy attack — a 3rd-party must prove it's acting on your behalf.
  • OIDC Federation (e.g. GitHub Actions, K8s) lets external IdPs assume roles without creating IAM users — keyless CI/CD.
D — Policy Types
identity-based Most common

Identity-based Policy

Attached to a user, group, or role. Grants what that identity can do across any resource. Can be AWS-managed, customer-managed, or inline.

Attach to: IAM User · Group · Role
resource-based Cross-account

Resource-based Policy

Attached to the resource (S3, KMS, Lambda, SQS…). Specifies who can access that resource. Enables cross-account access without assuming a role — the resource just trusts the external principal.

Attach to: S3 · KMS · SQS · Lambda · API GW
permissions boundary Max ceiling

Permissions Boundary

Sets the maximum permissions an identity-based policy can grant. Effective permission = intersection of identity policy AND boundary. A ceiling, not a grant. Used to safely delegate role creation.

Attach to: IAM User · Role (not groups)
org-level Org guardrail

Service Control Policy

Applied at the AWS Organizations level to an account or OU. Even account admins can't exceed what the SCP allows. Controls the maximum available permissions across the entire account — guardrails, not grants.

Attach to: Organization · OU · Account
session-level Per-session

Session Policy

Passed inline when calling AssumeRole or GetFederationToken. Further restricts permissions for that specific session only. Can't grant beyond what the role already allows.

Used with: sts:AssumeRole · GetFederationToken
legacy Mostly legacy

Access Control List

Legacy resource-level permissions for S3 buckets and objects. Predates resource-based policies. AWS recommends using bucket policies instead. Still used for some cross-account S3 object grants.

Attach to: S3 Bucket · S3 Object

F — Resource Hierarchy: How Each Cloud Structures Its Containers

AWS is account-centric with optional Organizations. Azure and GCP use strict hierarchical RBAC trees — permissions inherit downward. This matters because Azure/GCP let a role at a higher scope cover all children automatically.

graph TB subgraph AWS ["AWS — Account-centric"] direction TB a0["Root (Organizations)"] --> a1["Org Unit / OU"] a1 --> a2["Account"] a2 --> a3["Resources"] end subgraph Azure ["Azure — Subscription-based RBAC"] direction TB b0["Root Management Group"] --> b1["Management Group"] b1 --> b2["Subscription (billing unit)"] b2 --> b3["Resource Group"] b3 --> b4["Resources"] end subgraph GCP ["GCP — Project-based RBAC"] direction TB c0["Organization"] --> c1["Folder"] c1 --> c2["Project (billing unit)"] c2 --> c3["Resources"] end

Azure and GCP: permissions assigned at a higher scope inherit downward to all children. AWS does not have this natively — each account is isolated unless Organizations SCPs or resource policies are used.

G — Permissions Assignment: AWS vs Azure & GCP

Key structural difference: AWS bundles actions and resource ARNs in one JSON policy. Azure and GCP separate the role definition (what actions) from the scope (which resource) — they join at assignment time.

graph TB subgraph AWS ["AWS — Actions + Resources in one Policy JSON"] direction LR ia["Identity\n(User / Role)"] -->|"attach policy"| pa["Policy JSON\n{ Action: s3:GetObject\n Resource: arn:aws:s3:::my-bucket/* }"] pa -->|"grants access to specific resource"| ra["Resource"] end subgraph AzGcp ["Azure & GCP — Role separate from Scope"] direction TB ib["Identity"] --> assign["Role Assignment / IAM Binding"] rd["Role Definition\n{ Actions only\n no resource ARNs }"] --> assign sc["Scope\n(subscription / folder / project / resource)"] --> assign assign -->|"effective permissions on scope + children"| rb["Resource(s)"] end

AWS consequence: wildcard resource (*) in a policy grants access to all resources of that type — dangerous. Azure/GCP: over-permissive role at high scope = inherits to all children — equally dangerous, different shape.

H — AWS Policy Evaluation Logic

AWS checks policies in layers. An explicit Deny anywhere kills the request immediately. An Allow must survive all applicable layers to succeed.

flowchart TD REQ(["API Request"]) --> D1{"Explicit Deny\nin any policy?"} D1 -->|Yes| DENY(["❌ DENY"]) D1 -->|No| SCP{"SCP\nallows?"} SCP -->|No / not in org| DENY SCP -->|Yes| RBP{"Resource-based\npolicy Allow?"} RBP -->|Yes| ALLOW(["✅ ALLOW"]) RBP -->|No| IBP{"Identity-based\npolicy Allow?"} IBP -->|No| DENY IBP -->|Yes| PB{"Permissions\nBoundary OK?"} PB -->|Blocks| DENY PB -->|OK| SP{"Session\nPolicy OK?"} SP -->|Blocks| DENY SP -->|OK| ALLOW

Note: resource-based policy Allow can short-circuit to Allow without an identity-based policy — useful for cross-account S3 access. For same-account calls, both sides must allow.
